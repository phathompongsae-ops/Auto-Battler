# Asset Pilot Integration Contract v1

Status: Draft contract. Runtime integration remains CC-owned. Image generation remains ChatGPT-owned.

## Goal

Prove the complete asset and animation pipeline with exactly three pilots before mass-producing the game catalog:

- Archer: player hero and Shop/Bench/Board readability.
- Slime: small normal enemy and simple deformation-friendly motion.
- Golem: large miniboss silhouette, weight, hit readability, and death handling.

No complete Class 1 set, monster set, final cards, or final VFX should be produced until these three pilots pass in the live camera.

## Source policy

Use this order:

1. Exact user-approved existing file.
2. Downloadable asset with verified commercial-use-compatible license and complete provenance.
3. New asset generated by ChatGPT.

Coco must not generate images. A downloadable candidate is not canonical merely because it is free. Unknown-license, ripped, editorial-only, non-commercial-only, or watermarked files are forbidden.

Every downloaded candidate must record source URL, author/publisher, license, license evidence, download date, original filename, and SHA-256. Every exact file still requires user approval before canonical promotion.

## Canonical paths

- `assets/canonical/heroes/archer.png`
- `assets/canonical/monsters/slime.png`
- `assets/canonical/monsters/golem.png`

Battle assets, Shop-card portraits, and icons are separate outputs. A battle sprite must not be stretched into every UI role.

## Visual direction

All pilots use a readable stylized fantasy 3D/2.5D presentation, hand-painted materials, clean silhouette, transparent background, and three-quarter-front view.

Archer remains female, with an olive-green woodland hood/cape, bow, and quiver. Slime must remain readable at small size. Golem must read as heavy and larger than normal enemies without obscuring board state.

Runtime may flip X for facing. Do not create separate left/right art unless later evidence proves it necessary.

## Anchor and scale

The manifest owns scale and anchor. Feet/bottom-contact baselines must remain stable between animation states.

- Archer anchor: `(0.5, 0.92)`
- Slime anchor: `(0.5, 0.90)`
- Golem anchor: `(0.5, 0.94)`

Final runtime scale is intentionally validated in the live board camera, not hardcoded by the art file dimensions.

## Required animation states

Each pilot declares:

- Idle
- Move
- Attack
- Skill
- Hit
- Death

Target 12 FPS, accepted range 8–15 FPS. PNG sequence or sprite sheet is allowed. Frame count may differ per state, but feet baseline, canvas convention, and naming must remain consistent.

Fallbacks:

- Missing Move → Idle
- Missing Skill → Attack, then Idle
- Missing Hit → Idle
- Missing Death → Hit, then Idle

Fallback use must log once and must not crash the run. Existing runtime placeholders remain valid until the exact pilot file is approved.

## Naming convention

Recommended animation paths:

`assets/canonical/<group>/<asset-id>/<state>/<asset-id>_<state>_<frame>.png`

Examples:

- `assets/canonical/heroes/archer/idle/archer_idle_000.png`
- `assets/canonical/monsters/slime/attack/slime_attack_003.png`
- `assets/canonical/monsters/golem/death/golem_death_007.png`

Frames use zero-padded three-digit numbering starting at `000`. State names are lowercase.

## Technical requirements

- PNG with real alpha.
- No baked checkerboard or opaque background.
- Transparent bounds trimmed without cutting weapons, effects, hair, or death motion.
- No per-frame texture allocation in the game loop.
- Preferred battle frame size: at most 512 px on the longest side after export.
- Source/canonical review image may be up to 1024 px.
- Target maximum 1.5 MB per single pilot file and 12 MB for the complete three-pilot animation set.
- Target at least 20 simultaneous animated units on mobile without asset-caused frame collapse.

## Runtime acceptance criteria

1. Manifest resolves Archer, Slime, and Golem by stable IDs.
2. Missing files use a safe placeholder and log once.
3. Asset loading does not block the game permanently.
4. Idle, Move, Attack, Skill, Hit, and Death states can be requested through one shared interface.
5. X-flip preserves anchor and hitbox alignment.
6. Switching states does not change the feet baseline unexpectedly.
7. Death completes once and does not restore Idle afterward.
8. Hit can interrupt or overlay safely without leaking timers.
9. Shop-card and battle-asset paths stay separate.
10. Archer is readable in Shop, Bench, and Board contexts.
11. Slime is readable among multiple enemies.
12. Golem is visibly heavier/larger but does not hide essential tiles or units.
13. No texture, timer, or material leak after repeated waves.
14. No page error, NaN, Infinity, or missing-asset crash.
15. Mobile performance evidence is recorded before expanding production.

## Pilot review checkpoints

### Archer

Check female identity, woodland palette, bow/quiver silhouette, face readability, no overlap with Ranger/Sniper visual direction, and clear Board-facing orientation.

### Slime

Check compact silhouette, clear hit reaction, visible attack anticipation, and death without confusing it with a living idle blob.

### Golem

Check scale hierarchy, heavy anticipation, impact timing, hit flash compatibility, death fade compatibility, and visibility of nearby units/tiles.
